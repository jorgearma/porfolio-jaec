§ 01

Panchi-Bot

CASE STUDY
● landing ▶ demo en vivo

Bot de pedidos por WhatsApp para un restaurante o almacén. Gestiona el ciclo completo del pedido: desde el primer mensaje del cliente hasta el cobro y la entrega.

Arquitectura Flask en capas estrictas con procesamiento asíncrono de mensajes via Redis Queue, máquina de estados para el registro de usuarios, integración con Monei (pagos online) y validación de direcciones mediante geofencing + catálogo de calles.

Panel de operaciones con roles diferenciados: dashboard de administración, PWA de cocina, app de picking para almacén y app de reparto para repartidores. Todo con JWT por rol y estado sincronizado vía Redis.

Un restaurante gestionando pedidos por WhatsApp con capturas de pantalla, hojas de cálculo y llamadas. Sin trazabilidad. Sin escala. Sin sistema.

Panchi-Bot nació como solución real: automatizar el ciclo completo del pedido sin que el cliente instale nada y sin que el negocio abandone WhatsApp. La restricción era clara — el cliente ya vivía en WhatsApp y no iba a cambiar de canal.

El reto técnico no era trivial. WhatsApp no tiene sesión nativa: cada mensaje es un HTTP request aislado. El estado de una conversación multi-paso tiene que sobrevivir entre peticiones. Los pagos online necesitan webhooks firmados. Las direcciones necesitan validación geoespacial real, no solo geocodificación.

La solución es un sistema de producción completo: bot de WhatsApp, panel de administración, app de cocina, app de picking y app de reparto. Todo construido en Python con Flask, Redis y SQL Server. Sin dependencias de plataformas de terceros para la lógica de negocio.

Desafíos

01 · estado sin sesión nativa

WhatsApp no tiene estado

Cada mensaje entra como un HTTP request independiente. Para mantener un flujo multi-paso (registro, pedido, pago) sin perder contexto entre mensajes la solución fue Redis como capa de estado efímero con TTL por paso y deduplicación atómica mediante SET NX para evitar doble proceso en picos de carga.

02 · triple firma, tres esquemas

Tres webhooks, tres HMAC distintos

Twilio firma con SHA1 sobre la URL completa. Meta usa HMAC-SHA256 sobre el raw payload con el app secret. Monei tiene su propio esquema. Un error de verificación significa aceptar eventos falsos o rechazar pagos reales. Cada endpoint tiene su función de verificación propia; ninguno confía en el cliente.

03 · validación de dirección real

Google Maps no es suficiente

Una dirección puede geocodificarse perfectamente y aun así caer fuera del radio de reparto, o no existir en el catálogo de calles del municipio. La solución fue un pipeline de tres pasos: normalización regex → catálogo de 311 calles con búsqueda difusa (rapidfuzz) → geofencing con Shapely. El rechazo ocurre antes de que la cocina empiece a preparar.

04 · capa por capa sin atajos

Arquitectura sin deuda técnica

blueprints → controllers → managers → services. La regla es absoluta: nadie importa hacia arriba. Los 484 test cases no prueban el bot en global: prueban que cada capa cumple su contrato de forma aislada. Cuando algo falla, el stack trace apunta exactamente a dónde.
